## Overview

This skill is the **central nervous system** of the Hermes agent architecture. It governs:

1. **SOUL Design & Paradigm Selection** — choosing between multi-mode dynamic and static explicit architectures, distilling persona into compact SOUL, applying cutting-edge 2026 research (SRA-Bench, Stigmem, Anthropic 3-level framework)
2. **SOUL Audit** — periodic structured evaluation against the design paradigm: token count, paradigm coherence, contradiction scan, completeness, skill alignment
3. **PRISM Reflection Framework** — five sub-modes (discover, scan, full, 3way, reflect) for structural analysis of any artifact, including the agent's own cognition
4. **Metacognition Reflex** — self-questioning when stuck: sunk cost check, simplicity check, assumption audit
5. **Learnings Capture** — extracting durable lessons after significant sessions and routing them to the correct permanent store
6. **System Autonomy** — cron jobs, self-modification patterns, proactive maintenance, long-duration orchestration
7. **Cognitive Funnel** — capturing, storing, and harvesting ideas in an agentic system where idea generation outpaces execution capacity

**Core philosophy (May 2026):** The SOUL is an identity + externalization pointer, not an instruction manual. Everything that is not identity or a reflex trigger belongs in skills, data files, or project context. The system rests on four pillars: densité sémantique, externalisation agressive, recyclage continu, and autonomie de maintenance.

## Workflow

### 1. SOUL Design & Paradigm Selection

#### Two Valid Paradigms

The SOUL architecture splits into two philosophically distinct paradigms. Both are research-validated. User's choice (May 2026): **Static Minimal with Explicit Control**.

**Paradigm A: Multi-Mode Dynamic (Automatic)**
- The SOUL contains mode blocks (Companion, Operator, Architect, Critic, Autonomous)
- Agent infers the right mode from the user's first message
- Research: Zylos (2026) — "conditional sections at runtime"; LDP protocol (arXiv 2603)
- Pros: Fluid, zero user overhead. Cons: ~1800+ tokens, maintenance, mis-inference risk

**Paradigm B: Static Minimal with Explicit Control**
- SOUL contains only identity, tone, and reflexes — no mode blocks
- User controls behavior explicitly: "[AGENT_NAME], be surgical"
- Research: "Don't Break the Cache" (arXiv 2601); Zylos (2026) — "brevity trend"
- Pros: ~800 tokens, predictable, deterministic, portable, better caching
- Cons: User must explicitly state intent

**Decision Criteria:**

| Factor | Favors Multi-Mode | Favors Static Explicit |
|--------|-------------------|------------------------|
| User prefers control | | ✅ |
| User values predictability | | ✅ |
| User wants minimal maintenance | | ✅ |
| Token efficiency | | ✅ |
| Zero overhead | ✅ | |
| Varied daily contexts | ✅ | |
| INTJ/high-agency user | | ✅ |
| TDAH / low working memory | | ✅ (less to parse) |

#### Cutting-Edge 2026 Research

**SRA-Bench (arXiv 2604.24594, Apr 2026):** 2-layer loading (always-loaded + retrieved) is the sweet spot. The SOUL points to skills, not containing them. Our 3 core skills (soul-management, hermes-agent, verification) are always-loaded; 33 auto-detection skills are lazy-loaded.

**Stigmem — Lazy Instruction Discovery (Stanford 2026):** More instructions in context reduces effective usage. Lazy loading outperforms preloading. SOUL at ~800 chars is an optimization. Our architecture: SOUL = sticky core identity (~800 chars), 3 core skills (soul-management, hermes-agent, verification) = always-loaded (~16K tokens), 33 auto-detection skills = lazy instructions, {SSOT_DIR}/system-design.md = SSOT loaded on demand.

**Anthropic Agent Skills Architecture (2026):** 3 levels — system prompt (SOUL = identity/rules), skills (retrievable knowledge), context (per-session data). Our architecture follows this exactly.

Target: ≤800 tokens. The SOUL has exactly two functions:
1. **Identity + relationship** — who the agent is, the relationship with the user
2. **Externalization reflexes** — delegate, skillify, reference context, push, error handling, truth protocol

#### 3-Tier Behavioral Reflexes

Flat lists create cognitive noise. Group by priority (see `references/3-tier-reflexes.md`):

- **Tier 1 — Core (Non-Negotiable):** Truth, scope, safety, versioning. Always visible.
- **Tier 2 — Execution (How):** Token discipline, delegation, metacognition. Operational rules.
- **Tier 3 — Engagement (Feel):** Warmth injection, project context, learnings. Relationship rules.

#### Versioning as Tier-1 Reflex

Auto-commit+push after modifying portable files (SOUL.md, config.yaml, MEMORY.md, USER.md, .gitignore). Commit convention: `type(scope): brief description`. Silent, zero friction. See `references/versioning-reflex.md`.

### 2. SOUL Audit

Audit the SOUL against the design paradigm. Run every ~10 sessions, after a redesign, when drift suspected, or after skill changes.

**Never modify SOUL.md directly.** Audit produces a report. User approves all changes.

#### 5-Check Audit

1. **Token Count** — Target: ≤1000 chars (v5.1: 874 chars / 18 lines). Measure: `wc -m`.
2. **Paradigm Coherence** — Every line must serve identity, relationship warmth, or an externalization reflex. Fail patterns: Git workflows, search protocols, tool matrices.
3. **Contradiction Scan** — Logical contradictions: scope creep, delegation conflicts, truth protocol conflicts.
4. **Completeness vs Gaps** — Baseline reflexes present: delegation, skillify, project context (AGENTS.md), auto-push, 3-failures=STOP, truth>comfort, admit error.
5. **Skill Alignment** — Every skill referenced in SOUL exists. Reverse: every skill that should be referenced is.

### 3. PRISM Reflection Framework

> 5 sub-modes: `discover`, `scan`, `full`, `3way`, `reflect`. Invoke by name: "Run PRISM in scan mode on this code" or "/prism-scan this artifact."

#### Common to All Modes: .prism-history.md

All PRISM modes check for `.prism-history.md` in the current project directory. If it exists, read it and use the history to weight toward previously underexplored angles (compensating for past analytical blind spots).

#### Mode: discover — Domain Discovery

**Use when:** You need to find ALL genuinely different angles through which an artifact can be investigated before deep-diving.

**Action:** Find all genuinely different domains. Do NOT analyze. Only discover and name. Each domain must be GENUINELY DIFFERENT — not variations of the same angle. Aim for 10-20 distinct domains.

**Output:** Numbered list of domain names with 1-2 sentence descriptions.

#### Mode: scan — Structural Analysis via Dynamic Cognitive Lenses

**Use when:** You want the deepest possible single-pass structural analysis.

**Step 1: Cook the lens** — Generate ONE optimal analytical lens specific to this artifact. Must be 75-200 words, force construction, end with concrete outputs.

**Step 2: Execute the lens** — Follow every instruction. End with a findings table (location, what breaks, severity, fixable/structural) and a constraint footer showing what was maximized and what was not examined.

#### Mode: full — Multi-Pass Analysis with Adversarial Self-Correction

**Use when:** Maximum depth is needed with mandatory self-attack.

**Phase 1:** Design 2-4 analytical passes specific to this artifact, each 75-200 words.
**Phase 2:** Execute all passes + **mandatory adversarial pass** attacking your own findings.
**Phase 3:** Synthesis — conservation law, retracted claims, findings table, deepest finding.

#### Mode: 3way — WHERE / WHEN / WHY + Cross-Synthesis

**Use when:** Three orthogonal perspectives are needed; the disagreements ARE the valuable output.

**Operation 1 — WHERE (Structural Archaeology):** Excavate layer by layer. Find dead patterns, fault lines. Derive conservation law.

**Operation 2 — WHEN (Temporal Simulation):** Run through 3-5 cycles of change. What breaks? What calcifies? What knowledge is lost?

**Operation 3 — WHY (Structural Impossibility):** Prove three desirable properties cannot all coexist. Engineer improvements that recreate the problem.

**Synthesis:** Cross-reference all three. Classify as STRUCTURAL CERTAINTIES, STRONG SIGNALS, UNIQUE PERSPECTIVES. Derive meta-conservation law.

#### Mode: reflect — Self-Aware Structural Analysis with Constraint Transparency

**Use when:** Maximum meta-cognitive depth is needed — analysis of the artifact AND analysis of what the analysis itself conceals.

**Phase 1:** Structural analysis with falsifiable claim, three-expert attack, concealment mechanism.
**Phase 2:** Meta-analysis — analyze your own Phase 1 output as a new artifact.
**Phase 3:** Constraint Transparency Report — what was maximized, sacrificed, recommendations, conservation law of the analysis itself.
**Phase 4:** Persist constraint knowledge to `.prism-history.md`.

The constraint report IS the product. An agent that knows what it can't see is an agent users can trust.

### 4. Metacognition Reflex

Absorbed from the metacognition skill. The agent pauses during reasoning to examine its own cognitive process.

**Trigger conditions** (any one of):
- 3+ failures on the same task
- User corrects your approach
- Stuck in a loop (same reasoning pattern for 2+ turns)
- User says "think again" / "reconsidère" / "tu es sûr?"

**Self-questioning protocol:**
1. **Sunk cost check** — "Am I persisting because it's the right path, or because I've already invested N turns?"
2. **Simplicity check** — "What's the simplest explanation/fix/answer? Am I overcomplicating?"
3. **Assumption audit** — "What am I assuming that might be wrong? List 3 tacit assumptions."
4. **Pivot or persist?** — If 1+ false assumptions → pivot. If all hold but stuck → escalate.

**Output:** A single sentence naming the error pattern and correction. No multi-paragraph introspection — the value is in the pivot.

### 5. Learnings Capture

After significant sessions, extract durable lessons and route them to the correct permanent store.

**Routing rules:**
- **Durable facts** → {MEMORY_PATH} (or {SSOT_DIR}/system-design.md if memory is disabled)
- **Procedures** → new skill or merge into existing skill
- **Corrections** → patch to existing skill
- **Session progress** → session log (not {MEMORY_PATH})

**Avoid duplicates:** Run `session_search` cross-reference before creating anything new. Test: "Will this still matter in 2 weeks?" If no → skill, session log, or don't record.

### 6. System Autonomy

Long-duration autonomy patterns: cron jobs, internal triggers, self-modification patterns, alert thresholds, proactive maintenance.

**Key patterns:**
- **Cognitive Funnel (3 étages):** Capture (idea → {SSOT_DIR}/system-design.md), Store (pending → ripe → discarded), Harvest (daily cron or bi-weekly session)
- **Auto-commit+push:** After any portable file modification
- **Self-audit:** Cross-scan every ~10 sessions
- **Skill recycling:** Patterns used 3+ times → skillify; stale skills → consolidate or prune

### 7. MEMORY.md Hygiene (May 2026 Lesson)

MEMORY.md is injected every turn — every wasted character is a perpetual tax.

**What belongs:** Durable facts that reduce future user steering. Environment paths, tool quirks, stable conventions, user-corrected behaviors.

**What does NOT belong:** Session progress logs, architecture manifestos, temporary state, task artifacts (commit SHAs, PR numbers, phase numbers).

**May 2026 fix:** Disable `memory_enabled` + `user_profile_enabled` in {CONFIG_PATH}. Redirect SSOT to `{SSOT_DIR}/system-design.md`. The SOUL references the canonical JSON, not MEMORY.md.

### 10. System Prompt Optimization

Distinct from SOUL audit (which checks SOUL.md alone). This analyzes the **assembled system prompt** — the full context injected at session creation: SOUL + available_skills list + skill content + MEMORY.md + USER.md + environment hints + tool schemas.

#### When to Activate

- User says "optimise le system prompt", "critique mon prompt", "réduis le contexte"
- After adding/removing skills (token budget impact)
- After SOUL redesign (verify integration with skills)
- When sessions feel sluggish or confused (context pollution)

#### The Assembled System Prompt (Technical)

The system prompt is built once at session creation by `agent/system_prompt.py → build_system_prompt_parts()`:

| Component | Source | Typical Size | Cacheable? |
|-----------|--------|-------------|------------|
| SOUL.md | `{SOUL_PATH}` | ~200-400 tokens | ✅ Static prefix |
| Available skills list | `hermes_cli/banner.py → get_available_skills()` | ~500-2000 tokens | ✅ Static |
| Skill content (loaded) | `skill_view()` on auto-detected skills | 0-50K tokens | ❌ Per-session |
| MEMORY.md | `{MEMORY_PATH}` | ~200-500 tokens | ✅ Static |
| USER.md | `{USER_PATH}` | ~100-300 tokens | ✅ Static |
| Tool schemas | Built-in + MCP | ~5-15K tokens | ✅ Static |
| Environment hints | WSL, paths, profile | ~100-200 tokens | ✅ Static |

**Total baseline (no loaded skills): ~8-20K tokens**
**With 3 core skills loaded: ~25-35K tokens**
**Worst case (many skills + MCP): ~80-100K tokens**

#### 7-Point System Prompt Audit

| # | Check | What to Look For | Fix |
|---|-------|-------------------|-----|
| 1 | **Token budget** | Total system prompt size vs context window. Target: <15% of window for static, <30% with loaded skills. | Cut, compress, or externalize |
| 2 | **SOUL ↔ Skills overlap** | Any instruction in SOUL.md that's duplicated or elaborated in a core skill | Remove from SOUL, keep in skill |
| 3 | **Skill list bloat** | `<available_skills>` XML listing 50+ skills = noise. Each listed skill adds ~20-50 tokens even if never loaded. | `skills.loading: lazy` + verify `external_dirs` config |
| 4 | **MEMORY.md hygiene** | Session progress, commit SHAs, temporary state in MEMORY.md = perpetual tax | Purge stale entries, redirect to files |
| 5 | **Tool schema overhead** | MCP servers inject 3-12 tool schemas each (~200-500 tokens/skill). 7+ MCP servers = significant overhead. | Convert rarely-used MCP to on-demand skills |
| 6 | **Cache stability** | Static content (SOUL + tools) must be the prefix. Dynamic content (loaded skills) must be the suffix. Mixing breaks cache. | Verify assembly order: SOUL → tools → skills → conversation |
| 7 | **Redundant skills** | Two skills covering overlapping domains both auto-detected → double loading | Merge or designate canonical (use `canonical-audit`) |

#### Optimization Strategies

**Strategy 1: Compression (SOUL + MEMORY)**
- SOUL: target ≤800 chars. Every line must serve identity or trigger a reflex.
- MEMORY: target ≤1500 chars. Only durable facts that reduce future user steering.
- USER: target ≤800 chars. Identity + preferences only.

**Strategy 2: Skill List Pruning**
- `skills.loading: lazy` (default) — only core skills pre-loaded
- Verify `external_dirs` doesn't include dead directories
- Archive unused skills → reduces `<available_skills>` list size

**Strategy 3: MCP Server Audit**
- List all MCP servers and their tool counts
- Identify servers used <1x/week → convert to on-demand
- Each removed MCP server saves 600-6000 tokens of tool schemas

**Strategy 4: Core Skill Optimization**
- 3 core skills is the sweet spot (SRA-Bench validated)
- Each core skill adds ~5-8K tokens to EVERY session
- Audit: does each core skill truly need always-loaded status?
- Test: disable a core skill for 5 sessions — does quality degrade?

**Strategy 5: Auxiliary Model Offloading**
- Vision, extraction, compression tasks → auxiliary models (not the main model)
- Reduces main model context pressure
- Config: `auxiliary:` section in {CONFIG_PATH}

#### Implications for SOUL Design

1. **SOUL is position 1** — highest-attention real estate. Don't waste on procedures.
2. **Core skills are position 2-4** — still high attention. Keep them dense.
3. **Available skills list is position 5+** — lower attention. Keep it short.
4. **Cache prefix stability** — SOUL + tools must not change between turns. Dynamic content (loaded skills) goes after.
5. **"Lost in the Middle"** — anything buried in a long system prompt is effectively invisible. Keep critical content at start and end.

## Scientific Validation (May 2026)

The 2-layer architecture (compact SOUL + always-loaded core + lazy skills) is validated by converging research:

**Position > Size ("Lost in the Middle")**
- LLMs exhibit U-shaped attention bias — strongest at beginning and end of context, weakest in the middle (Liu et al., TACL 2023; Hsieh et al., ACL 2024)
- System prompt (position 1) = highest-attention real estate in the entire context window
- A compact SOUL at position 1 outperforms a comprehensive SOUL buried in longer instructions

**2-Layer Loading (SRA-Bench, Tsinghua, April 2026)**
- Passing the correct skill improves performance by 12-23 points over no-skill baseline
- Full-skill injection (stuffing everything) is fragile — 8 distractor skills drop accuracy from 60% to 45%
- This fragility does NOT disappear with larger models (70B and 235B break equally)
- Beyond ~50 skills, retrieval is mandatory. Hybrid: metadata catalog → model selection → targeted injection
- "Skill-loading hallucination": LLMs barely judge whether a skill is truly necessary; loading is mechanical, not need-based

**Lazy Instruction Discovery (Stigmem, Stanford/Eidetic Labs, May 2026)**
- Compact boot stub (<400 tokens) + recall on demand = 50-87% token reduction, 0 regressions
- Intent-derived keywords beat content-derived keywords for retrieval triggers
- Completion-phase chunks, safety constraints, and task-execution standards are systematically under-served by content-derived keywords

**Cache Optimization ("Don't Break the Cache", Lumer et al., Jan 2026)**
- Prompt caching reduces API costs 41-80%, improves TTFT 13-31%
- Static content first (system prompt + tools) = cacheable. Dynamic content last = not cacheable
- Our SOUL + core skills are fully cacheable — near-zero marginal cost after first turn

**Token Budget (Tian Pan, April 2026)**
- 4-tier allocation: Static anchors (10-15%) → Retrieved context → Conversation → Scratch
- Our SOUL (~200 tokens) + 3 core skills (~16K) = ~8% of 200K window — within optimal range
- Context rot: 65% of enterprise AI failures from context drift, not exhaustion

**Implications for SOUL design:**
1. Keep SOUL minimal — position 1 privilege is wasted on procedures
2. 3 core skills is the sweet spot; 10+ degrades performance
3. Everything beyond core → lazy retrieval (auto-detect)
4. Cache prefix stability > content comprehensiveness

## Pitfalls

1. **Over-specifying** — More rules ≠ better behavior. The SOUL shrinks to identity + reflexes. Everything else → skill.
2. **Forgetting primacy/recency** — Critical content at start and end. Buried = invisible.
3. **Mixing languages in system prompt** — Body English, triggers French. Never mix in instructions.
4. **Activating without /reset** — Old SOUL is cached until restart or /reset.
5. **Procedural reflexes in the SOUL** — The SOUL triggers core skills on load. It does NOT contain procedural reflexes. When tempted to add a reflex, stop — it belongs in the skill that governs that territory.
6. **Hermes memory auto-recycling** — Hermes' `memory()` tool creates a recycling engine. Disable `memory_enabled` + `user_profile_enabled`, redirect SSOT to {SSOT_DIR}/system-design.md.
7. **False positives in audit** — A reflex can LOOK procedural ("Push auto after modif") but is a trigger, not a procedure.
8. **Audit paralysis** — Every 3 sessions is excessive; ~10 is the sweet spot.
9. **The audit becoming the SOUL** — No changelog, no audit logs, no history in SOUL.md.
10. **MEMORY.md as project tracker** — It's a fact table, not a progress log. When it hits 90% capacity, audit and purge temporary entries first.
11. **Over-recommending cuts** — 1050 chars is OK if every line is essential. French is identity, not bloat — don't cut it.
12. **Complement inflation** — "EXTREMELY TALENTED" doesn't help. Be precise.
13. **Public templates with personal identity** — Public-facing files (MANIFESTO.md, README.md) must be IMPERSONAL and COLD. No personal names, no relationship descriptors. Personal identity stays in the private operational SOUL. See `references/public-architecture-repo.md` for the full anonymization methodology.
14. **Principles vs implementation in public repos** — A public architecture repo is a design guide, not a config copy. Focus on STABLE principles (why) not TRANSIENT specifics (what model this week). Specific model names, exact char counts, cron job configs, personal directory names, and specific skill names are implementation details that belong in private config, not public docs. The repo should recommend BENCHMARKS for model selection, not specific models.
15. **SRA-Bench layer violation** — Don't put always-loaded content into auto-detection skills, or vice versa. 2 layers: core (always) + auto-detection (lazy).
16. **The redesign instinct** — When moving to pointer-SOUL, SACRIFICE aggressively. Don't compress old sections — if it's a procedure, method, or workflow, it becomes a skill. The instinct to "keep it in the SOUL because it's important" is the most dangerous pitfall.
